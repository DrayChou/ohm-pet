# External pet formats

[简体中文](external-pet-formats.zh-CN.md) | English

OHM Pet treats imported packages as visual data. It never executes package JAR, EXE, DLL, SAORI, SHIORI, ghost, or script files.

## Installation

Copy a package directory or `.wlshm` file into the user-facing `pets/` directory, then choose `刷新宠物目录` from the tray menu. Package detection is recursive, so complete Shimeji and Ukagaka directory trees can be copied without flattening them.

## Supported formats

### Codex v2

```text
pet-id/
├── pet.json
└── spritesheet.webp
```

This remains the native runtime format.

### Shimeji-ee and compatible forks

Recognized layouts include:

```text
package/
├── conf/actions.xml
├── conf/behaviors.xml
└── img/<mascot>/shime*.png
```

and mascot directories containing `actions.xml` plus `shime*.png`. OHM Pet reads action names and `Pose Image` references, then maps standing, walking, running, jumping, dragging, falling, looking, and waving actions into the Codex/OHM state rows. Java code and behavior expressions are not executed.

### wl_shimeji

`.wlshm` packages with the `WLPK` header are supported. OHM Pet reads the embedded tar archive, `manifest.json`, `actions.json`, and QOI assets. Scripts and compiled expressions are ignored.

### Ukagaka and 伪春菜 shells

Recognized shell layout:

```text
shell-name/
├── descript.txt
├── surfaces.txt
└── surface*.png
```

UTF-8 and Shift-JIS metadata are accepted. Surface animations are normalized into idle, movement, jump, waiting, failed, and review states. This covers the visual shell layout used by DesktopPet's 茶兔 material and MaidChan.

### UkagakaW loose visual assets

Directories shaped as `Resources/Bitmap/` with multiple PNG images are loaded as a visual-only fallback pet. This is intended for incomplete engine repositories and prototype material that do not contain a standard SSP shell.

### csaori and other plugin repositories

csaori is a SAORI/plugin implementation rather than a pet shell. OHM Pet deliberately does not load or execute its DLLs. If a package contains a standard `shell/` directory, that visual shell is loaded; plugin-only packages correctly produce no pet.

## Action mapping

| Source semantics | OHM state |
| --- | --- |
| Stand, Sit, Idle, Surface 0 | Idle |
| Walk, Run, Dash, movement surface sequence | Running left/right |
| Wave, Greet, Look | Waving or review |
| Jump, Bounce, Dragged, Pinched | Jumping |
| Trip, Fall, failure expression surface | Failed |
| Sit, LookAtMouse, waiting expression | Waiting |

When semantic names are unavailable, deterministic surface fallbacks are used so the pet remains renderable. The adapter does not claim to reproduce the original dialogue or behavior engine.

## Costumes

Ukagaka `sakura.bindgroup<ID>.name` metadata and matching `animation<ID>.pattern...,bind,...` overlays are exposed as costumes when the referenced PNG exists.

Costumes can be changed from:

- the tray menu under `宠物换装`
- the context menu opened by right-clicking the pet

Only one costume per category is selected at a time, so one hat and one outfit can coexist. The selection is saved per pet. Packages without valid bind overlays do not show costume controls.

## Validation

Validate a copied package tree without launching the desktop window:

```bash
cargo run -p ohm-pet-core --example validate_external -- /path/to/pets
```

The validator decodes every pet and every advertised costume. It returns an error when a costume does not actually change the rendered idle frame.

Real third-party fixtures should stay under the Git-ignored `test-fixtures/external/` directory because artwork redistribution rights differ from source-code licenses. CI uses synthetic fixtures generated by the Rust tests.
